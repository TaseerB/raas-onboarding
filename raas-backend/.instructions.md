# RaaS Backend — Agent Coding Standards & Guidelines

## Project Overview

This is a **RADIUS-as-a-Service (RaaS) Management API** — a Spring Boot REST backend
that manages the configuration plane of a RADIUS stack (users, NAS clients,
certificates, RadSec tunnels, EAP profiles). It interfaces with FreeRADIUS via
configuration export and database sync.

Curriculum topics that map to features:
- Topic 1 — Networking (diagnostics/health API)
- Topic 2 — RADIUS AAA (user & NAS client management, AAA simulation)
- Topic 3 — PKI (Mini CA, certificate lifecycle)
- Topic 4 — RadSec (TLS tunnel config management)
- Topic 5 — EAP (EAP profile management)

## MANDATORY: Feature Context Protocol

**Before writing any code for a new feature or topic**, you MUST:

1. Create a feature context file at:
   ```
   raas-backend/docs/features/<topic-slug>/feature.md
   ```
   Example: `raas-backend/docs/features/topic-02-aaa/feature.md`

2. The `feature.md` MUST contain all of the following sections:
   ```markdown
   # Feature: <Name>
   ## Goal
   ## Endpoints (method, path, request body, response)
   ## Domain Entities & DB Schema (table name, columns, types, constraints)
   ## Flyway Migration (version number, filename)
   ## Service Responsibilities
   ## Dependencies (other features/services this feature requires)
   ## Edge Cases & Validation Rules
   ## Security Considerations
   ## Test Scenarios (happy path + at least 2 failure cases)
   ```

3. Only after the `feature.md` is reviewed and confirmed may you proceed to code.

## Layered Architecture Rules

1. **Controller** — HTTP boundary only. Validates input (Bean Validation), calls service, returns `ResponseEntity<ApiResponse<T>>`. No business logic. No JPA calls.
2. **Service** — Business logic, orchestration, transaction boundary (`@Transactional`). No `HttpServletRequest`/response objects.
3. **Repository** — Spring Data JPA interface. Custom queries via `@Query` (JPQL preferred over native SQL). No business logic.
4. **Entity** — JPA-annotated POJO only. No `@JsonIgnore` on sensitive fields — use DTOs to control serialization.
5. **DTO** — Plain Java record or `@Data` Lombok class. Used for API surface. Never expose entity classes directly in controllers.

## API Response Envelope

All endpoints **must** return `ApiResponse<T>`:

```java
// Success
ApiResponse.ok(data)           // 200
ApiResponse.created(data)      // 201

// Error (set via GlobalExceptionHandler)
ApiResponse.error(message)
```

Response shape:
```json
{
  "success": true,
  "data": { ... },
  "message": null,
  "timestamp": "2026-04-17T12:00:00Z"
}
```

## Security Requirements (OWASP Top 10)

1. **A01 Broken Access Control** — Every `/api/v1/**` endpoint requires a valid JWT Bearer token. Public: `/health`, `/swagger-ui/**`, `/v3/api-docs/**`, `/api/v1/auth/**`.
2. **A02 Cryptographic Failures** — Never store plaintext passwords or shared secrets. Use BCrypt (strength 12).  Never log passwords, secrets, or private key material.
3. **A03 Injection** — Use parameterized JPQL/native query (`@Query` with named params). Never concatenate user input into queries or shell commands.
4. **A07 Identification & Auth Failures** — JWT validation must verify signature, issuer, expiry. Use Spring OAuth2 Resource Server; do not write custom JWT parsing.
5. **A08 Software & Data Integrity** — Private keys returned via API must be transmitted only over TLS. Never return raw private keys in list endpoints.
6. **Input validation** — Use `jakarta.validation` annotations on all request DTOs (`@NotBlank`, `@Size`, `@Pattern`). Validate IP addresses with a regex constraint.
7. **No secrets in code** — All secrets (`jwt.secret`, DB password, etc.) come from environment variables or `application-*.yml` only. Never commit real credentials.

## Database / Flyway Rules

1. Migration files live at `src/main/resources/db/migration/`.
2. Naming: `V{n}__{snake_case_description}.sql` (double underscore).
3. Every table must have: `id BIGSERIAL PRIMARY KEY`, `created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`, `updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`.
4. Foreign keys must have explicit names: `CONSTRAINT fk_<table>_<col> FOREIGN KEY ...`.
5. Never edit an already-applied migration. Add a new version to fix it.
6. Test profile uses H2 in-memory (Flyway runs against H2 syntax — use standard SQL99; avoid PostgreSQL-specific syntax in migrations).

## Testing Standards

1. **Unit tests** — `src/test/java/...`. Test service layer with Mockito mocks. One test class per service class.
2. **Integration tests** — Use `@SpringBootTest` + `application-test.yml` (H2). Test controller layer via `MockMvc`.
3. **Test naming** — `methodName_whenCondition_thenExpected()`.
4. **Coverage** — Service layer: aim for 80%+ coverage on public methods. Edge cases (null input, not found, duplicate) must be tested.
5. **No `@Disabled` without a comment** explaining why and a linked issue.

## Lombok Usage

- Use `@Slf4j` on service and controller classes for logging.
- Use `@Data` on DTOs.
- Use `@Builder` on entities and response DTOs.
- Use `@RequiredArgsConstructor` on services/controllers (constructor injection, not `@Autowired`).
- Do **not** use `@Data` on JPA entities (causes issues with `hashCode`/`equals`). Use `@Getter @Setter @Builder @NoArgsConstructor @AllArgsConstructor` separately.

---

## Logging

- Use SLF4J (`@Slf4j`). Never use `System.out.println`.
- Log level: `DEBUG` for entered service methods, `INFO` for significant state changes, `WARN` for recoverable errors, `ERROR` for unhandled exceptions.
- Never log: passwords, private keys, secrets, JWT tokens, full stack traces at INFO level.

---

## General Code Style

- Prefer `Optional<T>` for nullable return values from repositories.
- `@Transactional(readOnly = true)` on read-only service methods.
- Use `record` for simple response DTOs when no mutation is needed.
- Avoid `null` returns from public methods — use `Optional` or throw a typed exception.
- Max method length: ~30 lines. Extract to private helpers if longer.
- All public API surface must have `@Operation` and `@Tag` from SpringDoc.
